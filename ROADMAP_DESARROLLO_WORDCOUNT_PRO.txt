================================================================================
1. PROPOSITO Y ALCANCE DE ESTE ROADMAP
================================================================================
El Plan de Negocios responde "qué construir y por qué". El Plan de
Arquitectura responde "cómo construirlo técnicamente". La Lógica de Negocio
responde "qué reglas exactas debe cumplir el código". Ninguno de los tres
responde, día a día, "qué hago esta semana y cómo sé que terminé bien".

Ese es el propósito de este Roadmap: convertir las tres fuentes en un
calendario operativo, sprint por sprint (ciclos de 2 semanas, según la
recomendación metodológica del Plan de Negocios, sección 16.2), de modo que
un equipo de 1 a 2 personas (la realidad descrita en el Plan de Negocios,
sección 11) pueda ejecutar el proyecto sin perder de vista ni la fecha de
entrega académica, ni la integridad arquitectónica, ni las reglas de negocio
que protegen el modelo de monetización.

Alcance temporal: desde la Semana 0 (preparación) hasta el cierre del Año 1
(Mes 12) con detalle de sprint, y desde el Año 2 en adelante con detalle
trimestral (estratégico, no de sprint), ya que el propio Plan de Negocios y
el Plan de Arquitectura tratan el Año 2 como "visión de escalabilidad" sin
comprometerse a fechas exactas.


================================================================================
2. PRINCIPIOS Y METODOLOGIA DE TRABAJO
================================================================================
2.1 Cadencia: sprints de 2 semanas con tablero Kanban (Trello o Notion, según
    la recomendación del Plan de Negocios, sección 16.2). Cada sprint cierra
    con una entrega funcional, no con código a medias.

2.2 Trazabilidad obligatoria: todo commit, pull request y test relevante
    debe referenciar el código de regla de negocio que protege (formato
    BR-XXX, según la Lógica de Negocio, sección 1), por ejemplo
    `test_BR014_limite_5000_palabras_plan_gratuito`.

2.3 Regla de oro heredada de la Lógica de Negocio: ninguna regla de negocio
    vive en el router/endpoint ni en el componente de React; toda regla vive
    en la capa de Servicio del backend, salvo el caso explícitamente
    autorizado por el Plan de Arquitectura (Módulo 1 gratuito calculado en
    cliente, ADR-004).

2.4 CI obligatorio desde el primer sprint con código: ruff + mypy + pytest en
    backend, eslint + vitest en frontend (Plan de Arquitectura, sección 9.4).
    Ningún sprint se considera cerrado si el pipeline de CI está en rojo.

2.5 Revisión de ADRs al inicio de cada Fase: tal como indica la conclusión
    del Plan de Arquitectura (sección 15), antes de iniciar cada nueva Fase
    (1→2, 2→3, 3→4) el equipo debe releer la sección 13 (ADRs) y la sección
    19 de la Lógica de Negocio (Decisiones Pendientes) y resolver lo que
    corresponda antes de avanzar.

2.6 Cada sprint de este roadmap incluye seis campos fijos: Objetivo,
    Backend, Frontend, Infraestructura/DevOps, Reglas de negocio (BR-XXX),
    y Entregable + Criterio de aceptación. Esto evita que el equipo confunda
    "trabajé en algo" con "completé algo verificable".

4. SPRINT 0 — PREPARACION DEL ENTORNO (SEMANA PREVIA AL MES 1)
================================================================================
Objetivo: eliminar toda fricción de entorno ANTES de que empiece a correr el
cronograma de 12 meses del Plan de Negocios.

Tareas (Plan de Arquitectura, sección 9.1):
  - Instalar Git para Windows, configurar `.gitattributes` (`* text=auto
    eol=lf`) en la raíz del repositorio para evitar conflictos CRLF/LF.
  - Instalar Python 3.12+ (marcar "Add Python to PATH") y Node.js LTS.
  - Instalar Docker Desktop para Windows con backend WSL2 habilitado (vía
    para Postgres/Redis idénticos a producción).
  - Crear repositorio Git con la estructura de carpetas `backend/` y
    `frontend/` definida en el Plan de Arquitectura, secciones 4.2 y 6.2.
  - Crear `docker-compose.yml` inicial (servicios `db` y `redis`) y
    `.env.example` en ambas carpetas.
  - Instalar Visual Studio Code con extensiones: Python, Pylance, ESLint,
    Prettier, Tailwind CSS IntelliSense, Docker, GitLens.
  - Configurar tablero Kanban (Trello/Notion) con las columnas: Backlog /
    Sprint Actual / En Revisión / Hecho.
  - Crear cuentas free-tier: Vercel, Railway, Supabase, Cloudflare, Sentry,
    UptimeRobot (todas mencionadas en el Plan de Negocios y el Plan de
    Arquitectura como base de infraestructura de costo casi nulo).

Entregable y criterio de aceptación: un desarrollador nuevo puede clonar el
repositorio, ejecutar `docker compose up -d db redis`, levantar el backend
con `uvicorn app.main:app --reload` y el frontend con `npm run dev`, y ver
`http://localhost:8000/docs` y `http://localhost:5173` funcionando, en menos
de 30 minutos siguiendo solo el README.

13. DEFINICION DE "HECHO" (DEFINITION OF DONE) GENERAL
================================================================================
Una tarea, sprint o fase de este roadmap se considera HECHA solo si cumple
TODO lo siguiente:

  1. El código pasa el pipeline de CI completo (ruff, mypy, pytest en
     backend; eslint, vitest en frontend) sin excepciones ni tests
     deshabilitados.
  2. Toda regla de negocio implementada tiene al menos un test que
     referencia su código BR-XXX en el nombre.
  3. Ninguna regla de negocio vive en el router/endpoint ni en el
     componente de React (principio 2.3 de este roadmap).
  4. El entregable fue verificado manualmente end-to-end por al menos una
     persona distinta a quien lo codificó (o, en equipo de 1 persona, tras
     un descanso de al menos un día antes de la verificación).
  5. La documentación relevante (README, ADR, comentarios de regla BR-XXX)
     quedó actualizada en el mismo sprint, no "para después".

======================================================================
NUEVAS ACTUALIZACIONES Y MEJORAS IMPLEMENTADAS (VERSIÓN ACTUAL)
======================================================================

Las siguientes funcionalidades han sido integradas exitosamente en la plataforma, modificando la arquitectura, logica y negocio de la misma para mejorar la experiencia del usuario y las capacidades tecnicas:

1. Unificacion de Roles y Diseno Universal:
   - Se ha eliminado la estricta separacion entre 'Estudiante' y 'Profesor' en la pantalla de inicio de sesion. Ahora existe un solo tipo de usuario general ('general') para simplificar el acceso.
   - El diseno se ha redisenado con una estetica Premium (Fondo Morado Pastel, fuentes modernas y sin botones negros).

2. Modulo 'Mis Documentos (Nube)':
   - Se creo un panel visual especifico en la barra de navegacion para que los usuarios puedan ver y organizar sus documentos.
   - El sistema guarda todo automaticamente cada 3 segundos en la base de datos y lo presenta en forma de 'Tarjetas' interactivas desde donde se puede recuperar el texto con un clic.

3. Dashboard para Desarrolladores y Consola API:
   - Se integro un entorno de pruebas (Playground) 100% funcional.
   - Incluye una Calculadora de Ahorro en tiempo real que compara los costos de consumo frente a OpenAI.
   - Permite la generacion y revocacion de API Keys seguras para simulacion de entornos de produccion.

4. Motor de Conteo Universal (Unicode):
   - El algoritmo de deteccion de palabras se ha actualizado con estandares Unicode.
   - Ahora detecta y cuenta con absoluta precision caracteres alfanumericos, tildes, enies, y vocablos extranjeros.

5. Persistencia de Sesion y Modo Libre:
   - Persistencia (localStorage): La sesion de usuario sobrevive a recargas de pagina (F5) o cierres de pestana, mejorando la usabilidad.
   - Modo Libre de Pruebas: El backend ha sido configurado para aceptar cualquier contrasena durante la fase de desarrollo, facilitando pruebas simultaneas entre el equipo.
